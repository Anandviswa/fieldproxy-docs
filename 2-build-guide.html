Build Guide

Follow this day by day. Open it when you're building, not before.

Prerequisites: Read Getting Started first. Keep Quick Reference open beside you.

Day 1 — Understand the Machine (Web + Mobile)

Morning (3-4 hours): Read and explore

  1. Read the fieldproxy-app-builder skill document end to end. Don't skim.
  2. Understand these core concepts (both web and mobile):
    • App shell: type: "web" or type: "mobile" → contains views/screens → each has ui + data
    • Widgets/Elements: Every visible element has a type (container, text, button, input, row, etc.)
    • Data queries: SQL in the data object, results accessible via templates
    • Template expressions: {{query.data.[0].field}} binds live data to widgets
    • onclick actions: Numbered action chains that execute sequentially
  3. Key template patterns to memorize (work on BOTH platforms):
    PatternWhen to use
    {{query_name.data.[0].field}}Display a value from a query (first row)
    {{query_name.data.[index].field}}Inside a list — current item's field
    {{arguments.param_name}}Value passed from another view via navigation
    {{input_data.element_id.0}}Value the user typed into an input
    {{input_data.element_id.[index]}}Input value inside a list context
    {{rowData.field}}Inside a table — current row's field (web)
    <!getUserId()!>Current user's ID (mobile)
    <!getLocation().latitude!>Device GPS latitude (mobile)

Afternoon (3-4 hours): Map existing apps — one web, one mobile

  1. Web: Open the Safegroup web app. Open enquiry_detail_view.json side by side. Find:
    • Where is the sidebar defined?
    • Which SQL query powers the detail card?
    • How does the status badge get its color?
    • What happens when you click “Quick Actions”? (trace the onclick chain)
  2. Mobile: Open one of these side by side with the JSON:
    • projects/EFS/cleaner app/cleaner_home.json — home screen with clock-in
    • projects/safegroup/Safegroup Web App/Mobile App/visit_list_screen.json — visit list
    Find:
    • How is the data query structured? (same data object as web?)
    • How does list item navigation work?
    • Where is onclick placed — on the row or on a wrapping container?
  3. Read fieldproxy-widget-rules (34 rules) and fieldproxy-mobile-spacing (10 mobile patterns).
  4. Skim the first 20 entries in fieldproxy-error-log.

End of Day 1: You can verbally explain how a web view and a mobile screen are structured, and what they share vs what differs.

Day 2 — Build the Skeleton (Web + Mobile)

Morning: Database + Web app shell

  1. Create your project folder: 
    projects/{your_project}/
├── Client Docs/
├── Database/
│   └── schema.sql
├── Web App/
│   └── Screens/
│       └── dashboard/
├── Mobile App/
└── Calls/
  2. Create the database schema — use correct FieldProxy datatypes: 
    CREATE TABLE my_items (
  id            INTEGER PRIMARY KEY GENERATED ALWAYS AS IDENTITY,
  title         TEXT,
  status        TEXT DEFAULT 'open',
  priority      TEXT DEFAULT 'medium',
  assigned_to   TEXT,
  is_active     BOOLEAN NOT NULL DEFAULT TRUE,
  delete        BOOLEAN NOT NULL DEFAULT FALSE,
  deletedat     TIMESTAMP WITH TIME ZONE,
  createdat     TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP,
  updatedat     TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX idx_my_items_status ON my_items(status);
CREATE INDEX idx_my_items_is_active ON my_items(is_active);

    Schema conventions: TEXT (never VARCHAR), GENERATED ALWAYS AS IDENTITY (never SERIAL), TIMESTAMP WITH TIME ZONE (never bare TIMESTAMP), DOUBLE PRECISION for decimals. Standard columns: is_active, delete, deletedat, createdat, updatedat. Always add indexes.

  3. Write the web app JSON shell BY HAND: 
    {
  "type": "web",
  "title": "My First App",
  "app_id": "YOUR_APP_ID",
  "default_view": "dashboard_view",
  "element_id": "app_root",
  "views": {
    "dashboard_view": {
      "ui": [],
      "data": {},
      "label": "Dashboard",
      "view_id": "dashboard_view"
    }
  }
}
  4. Add a sidebar with 3 navigation items. Add a data query: 
    "data": {
  "stats": "SELECT COUNT(*)::TEXT as total, COUNT(CASE WHEN status='open' THEN 1 END)::TEXT as open_count FROM my_items WHERE delete = false"
}

Afternoon: Mobile app shell (same data)

  1. Build the mobile home screen — same database, same query: 
    {
  "type": "mobile",
  "title": "My Mobile App",
  "app_id": "YOUR_MOBILE_APP_ID",
  "element_id": "mobile_root",
  "ui": [
    { "type": "container", "element_id": "page_wrapper", "padding_top": 16, "padding_left": 16, "padding_right": 16, "padding_bottom": 16, "ui": [] }
  ],
  "data": {
    "stats": "SELECT COUNT(*)::TEXT as total, COUNT(CASE WHEN status='open' THEN 1 END)::TEXT as open_count FROM my_items WHERE delete = false"
  }
}
  2. Add info card + container-based buttons. See Quick Reference for mobile patterns.

Compare: Open both JSONs side by side. data queries are identical, layout differs.

End of Day 2: Both a web app and mobile app showing live data from the same database.

Day 3 — Build the List (Web + Mobile)

Morning: Web table/list view

  1. Create items_list_view in Web App/Screens/items/.
  2. Add search-enabled data query.
  3. Build a table widget with search input and status dropdown filter.
  4. Add row click → navigation to detail view with arguments.
  5. Add “Add New” popup: execute_query → toast → close_popup → update_data.

Afternoon: Mobile list screen (same data)

  1. Create list_screen.json in Mobile App/. Same SQL query.
  2. Build a type: "list" — each item is a card container with:
    • onclick on the container (NOT the row)
    • Badge wrapped in main_axis_size: "min" parent
    • expanded: true on title text

Compare: Web uses table widget, mobile uses list with card items.

End of Day 3: Working list on both platforms with search and navigation.

Day 4 — Build the Detail + CRUD (Web + Mobile)

Morning: Web detail page

  1. Create item_detail_view. Query: SELECT * FROM my_items WHERE id = {{arguments.item_id}}
  2. Header row: back button + title + spacer + status badge
  3. Two-column card: info left (width: 6.5), actions right (width: 4.5)
  4. Edit popup + Delete with confirm → execute_query → toast → navigation
  5. Add tab_group with “Details” and “Activity” tabs.

Afternoon: Mobile detail screen

  1. Same query. Stacked cards instead of columns.
  2. Status badge with main_axis_size: "min" wrapper.
  3. Delete uses navigation with back: true (not navigation_type: "view").

End of Day 4: Full detail on both platforms with CRUD.

Day 5 — Build Forms + API Actions (Web + Mobile)

Morning: Forms

  1. Web: Popup form — text inputs, dropdowns, confirm chain.
  2. Mobile: Full-screen form — same fields, submit → execute_query → toast → back: true.
  3. Both: API actions work identically on both platforms: 
    { "type": "api_call", "url": "https://...", "method": "POST", "body": {...}, "element_id": "..." }

Afternoon: Dashboard KPIs

  1. Web: grouped_counter_1 with width: 12, columns: 4, element_count: 4, isBackground: true, customAspectRatio: 1.3.
  2. Mobile: Info card pattern for the same KPIs.

End of Day 5: Complete forms, APIs, and dashboards on both platforms.

Day 6 — Self-Debug + Mobile Deep Dive

Morning: Self-debug both apps

  1. Web — check against 34 widget rules:
    • No width on column or row?
    • border_color + border_width (not CSS border)?
    • space_widget in rows has alignment: "horizontal"?
    • Every element has unique element_id?
    • expanded: true only on text?
    • Data queries SELECT-only?
  2. Mobile — check against 10 mobile patterns:
    • Colored badges wrapped with main_axis_size: "min" parent?
    • Dynamic text has expanded: true?
    • Tappable rows wrapped in container with onclick?
    • Popup close uses back: true?
    • No color: "transparent"?
  3. Fix everything. This is the most valuable exercise of the week.

Afternoon: Mobile-specific features

Study existing apps:

  1. Image input + camera: projects/EFS/cleaner app/create_issue.json
  2. Multi-section checklist: projects/safegroup/.../ASHP Commissioning/ — 9 sections
  3. Location capture: <!getLocation().latitude!> in attendance screens
  4. Signature input: ashp_s9_signatures.json

End of Day 6: Both apps debugged. Mobile-specific features understood.

Day 7 — Polish + Present

Morning: Final touches

  1. Database schema review — indexes, standard columns present?
  2. Web ↔ mobile consistency — same queries, same CRUD logic, matching dropdown values?
  3. Add empty states if missing.

Afternoon: Present (20 min)

  1. Demo both apps (8 min) — same user flow on web then mobile
  2. Walk through shared vs different JSON (5 min)
  3. What you learned (4 min)
  4. Questions (3 min)

Project Folder Structure

projects/{client_name}/
├── Client Docs/
├── Calls/
├── Database/                       ← Shared by web AND mobile
│   ├── schema.sql
│   └── seed_data.sql
├── Web App/
│   ├── Screens/{module}/
│   ├── Queries/
│   └── Workflows/
└── Mobile App/
    ├── homescreen.json
    └── {module}/

Rules: Database is shared. Web screens in Web App/Screens/{module}/. Mobile screens in Mobile App/{module}/. One folder per entity.
