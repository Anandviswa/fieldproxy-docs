Getting Started

Read this end-to-end on Day 1. It takes about 30 minutes.

Welcome to FieldProxy

FieldProxy is a Field Service Management (FSM) platform — the system that companies use to manage their entire field operations: jobs, technicians, checklists, schedules, invoices, and customer communication.

We're not a traditional FSM tool. We're building the most powerful agentic FSM platform in the world.

How We Got Here

FieldProxy went through three eras:

  1. Drag & Drop Era — A visual builder. Worked for simple setups, but couldn't handle complex field operations. Customisation hit a wall.
  2. Everything Became Configuration — We made a radical choice: every screen, workflow, field, view, and business rule is defined as structured, machine-readable JSON config. The entire application is a definition file.
  3. Then AI Arrived — AI models are exceptionally good at reading and writing structured config. Our architecture turned out to be exactly what AI needs to not just assist — but actually operate.

Key insight: We didn't plan to build an AI-native platform. We built for flexibility — and flexibility is what AI runs on.

The Template Account Model

This is critical to understand. It changes how you think about every project.

The old way was to build from scratch for every client. Weeks turning into months. Expensive, slow, doesn't scale.

The new way is one deep, mature Template Account — a production-ready FSM with agentic workflows on top. Every new client starts from this template.

Clone Template → Configure on Top (AI agent helps) → Live in 1-2 weeks

Why this matters for you:

  • You don't start from a blank canvas. Every project starts from the template.
  • Every deployment feeds back. What you build for Client A makes the template smarter for Client B.
  • AI agents handle customisation. Need a new workflow? Tell the agent.
  • One product to demo, one story to sell. Every sales call shows the same mature platform.

Every client makes the template smarter. Every rollout gets faster. This is a compounding loop.

The Technical Foundation

FieldProxy lets you build complete web applications AND mobile applications purely through JSON configuration. No HTML, no CSS, no React, no Flutter. You write JSON, bind it to SQL queries, and the platform renders it.

JSON (layout + widgets) + SQL (data) = Working app (web or mobile)

Both web and mobile apps share the same foundation:

  • Same SQL database — both read and write to the same tables
  • Same data binding — both use template expressions to display query results
  • Same action system — both use numbered onclick chains for user interactions
  • Same API capability — both can trigger external APIs via onclick actions

The difference is in layout and rendering — web uses a 12-column grid with sidebar navigation, mobile uses full-width layouts with scroll-based navigation.

Each day in Week 1 you'll build the same feature on both web and mobile — so you learn the differences by direct comparison. Follow the Build Guide day by day.

The Three Concepts

1. Skills — The universal rules

Skills are the rules of the FieldProxy framework. They apply to every view — web AND mobile — every project, every team member.

  • “Buttons use grid width 1-12, not fixed_width”
  • “space_widget inside a row needs alignment: horizontal”
  • “onclick keys must be pure numeric strings: 1, 2, 3”
  • “Row onclick doesn't work on mobile — wrap in container”

Breaking a skill rule = your view breaks or looks wrong. No exceptions. Think of skills as traffic rules. Everyone follows them.

2. Recipes — Other people's working views

Recipes are real views that someone on the team built and extracted as templates:

  • “Here's how Anand built an enquiry detail page with email threads” (web)
  • “Here's how the EFS cleaner home screen was built with clock-in/out” (mobile)
  • “Here's how the Safegroup ASHP checklist works across 9 sections” (mobile)

You can follow a recipe exactly, adapt it, or ignore it entirely. Think of recipes as route suggestions.

3. Your creativity — The destination

Skills keep you safe (no layout bugs). Recipes save you time (don't start from scratch). But what you're building and how it looks? That's your design.

Skills  = Prevent bugs      (everyone follows)
Recipes = Save time         (pick and adapt, or ignore)
You     = Design the product (your vision)

When NOT to Use AI

This is critical. Read this before your first day.

SituationWhy you should NOT use AI
You don't understand what the JSON output meansYou can't debug what you can't read
You can't describe what you want in specific words“Make it look nice” produces garbage
You're learning a new widget type for the first timeBuild it by hand first. Break it. Fix it.
You're debugging a layout issueRead the rules yourself, find the violation
The view is simple enough to write manuallyOver-relying on AI kills your muscle memory

The progression

Week 1-2:  Build by hand. Read skills as DOCUMENTATION, not as AI tools.
Week 3-4:  Use AI for repetitive parts. Build complex parts by hand.
Month 2+:  Use AI freely — you know enough to verify and fix the output.

The danger sign: If you can't look at a generated JSON and spot a mistake — you're using AI too early. Understanding first, speed second.

Top 10 Most Repeated Mistakes

Every bug in the error log was caused by a real person on this team. Reading it saves you from the same mistakes.

  1. CSS border doesn't work — use border_color + border_width separately
  2. icon_button uses bg_color/color/size — NOT background_color/icon_color
  3. input_type: "search" doesn't exist — use "text"
  4. visibility_condition on inputs is ignored — wrap in container
  5. expanded: true only works on text — not containers/buttons
  6. Container bg uses color — NOT background_color
  7. show_if doesn't exist — use visibility_condition
  8. fixed_height doesn't exist — use height
  9. type: "divider" doesn't render — use container with height:1
  10. input_data keyed by element_id — NEVER by field_name

How to Use Claude Code

Claude Code is the AI assistant integrated into VS Code. It reads CLAUDE.md automatically — so it knows our project rules.

The Golden Rule

Claude Code reads the skills and error log before writing any JSON. This is enforced in CLAUDE.md. If you ask Claude to build a view, it will:

  1. Load fieldproxy-widget-rules.md (34 rules)
  2. Load fieldproxy-error-log.md (86+ past bugs)
  3. THEN generate the JSON

What Claude is Good At

TaskExample prompt
New screen“Build a list view for tickets with search, status filter, and a table showing title, status, assigned_to, created_date”
Add widget“Add a status badge pill to the header row in visit_detail_screen.json”
SQL queries“Write a query that counts open/closed/overdue tickets grouped by assigned_to”
Fix layout“The columns are overlapping in the facility list card — address text runs into the actions”
Onclick chains“When the user clicks Save, INSERT into tickets, show a success toast, close the popup, and refresh the list”

What Claude is BAD At (During Your First 2 Weeks)

  • “Build me the whole app” — you won't understand the output, can't debug it
  • “Fix this” (without symptoms) — vague prompts produce vague fixes
  • Letting it design the UX — it follows patterns, not design briefs. Check with Afrin.
  • Trusting output without reading it — YOU must verify every time

Verification Checklist

Run this after Claude generates JSON:

  • Every element has a unique element_id?
  • No CSS border — uses border_color + border_width?
  • expanded: true only on text widgets?
  • input_data references use element_id, not field_name?
  • space_widget in rows has alignment: "horizontal"?
  • Container backgrounds use color, not background_color?
  • Data queries are SELECT-only? (mutations in onclick → execute_query)
  • Mobile: Row onclick wrapped in container? Badges wrapped with main_axis_size: "min"?

Web vs Mobile

Both platforms share the same SQL database and action system. The differences are in layout and rendering.

What's Shared

  • Database schema — same tables for both web and mobile
  • SQL queries — often identical or very similar
  • Business logic — same INSERT/UPDATE/DELETE in onclick chains
  • Template expressions — same {{query.data.[0].field}} syntax
  • Action types — same execute_query, toast, navigation, confirm, api

What's Different

  • Layout approach — web uses grid widths, mobile uses full-width + padding
  • Navigation — web uses sidebar + view switching, mobile uses screen-to-screen
  • Mobile-only features — GPS location, camera/image input, signature capture, barcode scanning
  • Rendering quirks — mobile has specific rules about row onclick, badge stretching, dividers, transparent colors

You're not just writing JSON. You're building the screens that field technicians, managers, and operations teams use every day to run their businesses.

Every view you build:
