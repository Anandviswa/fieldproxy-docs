1. Overview

What is Fieldproxy Mobile

Fieldproxy Mobile is a Flutter-based, low-code mobile application platform. Screens (“views”) are defined entirely in JSON and rendered natively on iOS and Android. The same JSON-driven philosophy as the web platform is used, but the mobile runtime adds native capabilities: camera, GPS, barcode scanning, offline SQLite evaluation, push notifications, and background geolocation.

How JSON Configs Drive Native Mobile UI

Every view is a JSON object inside the views dictionary. When a screen is navigated to, the ViewController loads the view definition, runs SQL queries listed in data{}, fires onload{} actions, then passes the ui[] array to getChildren() which recursively creates Flutter widgets.

How Data Queries Work

Each key in data is a SQL string. The runtime calls the backend API with that SQL, stores results in viewData[key], and makes them available via {{key.data.[index].column_name}} templates.

"data": {
  "jobs_list": "SELECT id, title, status FROM jobs WHERE assigned_to = '<!getUserId()!>'",
  "customer_info": "SELECT * FROM customers WHERE id = {{arguments.customer_id}}"
}

onload Actions

Executed once after all data queries complete. Uses the same ordered action map format as onclick.

"onload": {
  "1": {
    "type": "update_value",
    "element_id": "date_field",
    "value": "<!getCurrentDate()!>"
  }
}

Back Confirmation (🆕 NEW)

Intercepts the system back gesture / button with a confirmation dialog.

"back_confirmation": {
  "enabled": true,
  "title": "Confirm Navigation",
  "message": "Are you sure you want to go back? Unsaved changes will be lost."
}

To dismiss programmatically after a successful save, add "disable_back_confirmation": true to an execute_query or api action.

11. Conditional & Logic System

visibility_condition

SQL expression string. The element is hidden when the expression evaluates to 0 (or falsy). The expression is wrapped in SELECT ... AS result and run against local SQLite.

"visibility_condition": "'{{input_data.show_signature.[0]}}' = 'yes'"
"visibility_condition": "{{job_detail.data.[0].status}} != 'completed'"

disabled_condition

When the expression evaluates to 1, the element is greyed out and non-interactive.

"disabled_condition": "'{{input_data.is_locked.[0]}}' = 'true'"

required_condition

Element is required only when this expression evaluates to 1.

"required_condition": "'{{input_data.service_type.[0]}}' = 'warranty'"

Action Condition Guards

Any action in the onclick map can be conditionally skipped.

{
  "type": "execute_query",
  "condition": {
    "type": "expression",
    "condition": "'{{input_data.notes.[0]}}' != ''"
  },
  "query": "INSERT INTO notes ..."
}

12. Mobile-Specific Features

Camera & Image

The image input type has three upload_mode values:

  • "DEFAULT" — presents a dialog letting user choose Camera or Gallery
  • "CAMERA" — goes directly to camera
  • "GALLERY" — opens device photo library

Internal Camera vs External:

  • By default uses the in-app CameraView (Fieldproxy custom camera)
  • Can be configured to use the device’s native camera app

Multi-Image Capture: When is_multi: true and camera mode, the app uses MultiCaptureCameraView to capture multiple images in one session.

Watermarking: When isWatermark: true, the runtime burns the following into the image before upload:

  • "timestamp" — current date/time text
  • "userId" — current user ID
  • "location" — current GPS coordinates

OCR: When isOcr: true, the captured image is processed by Google ML Kit text recognition. With ocrType: "pan", it parses PAN card fields automatically.

Auto-Compression: Images over 2MB are automatically compressed to 70% quality at 1920x1080 resolution before uploading.

Upload Progress: Upload status is tracked per file (states: uploading, completed, error). Users can see progress overlays and retry failed uploads.

GPS / Location

  • Background geolocation via flutter_background_geolocation
  • Current location is always available via <!getLocation().latitude!> / <!getLocation().longitude!>
  • LocationInput opens an interactive map where user pins their location
  • Geofencing is configurable via fenceRadius property

Barcode / QR Scanning

  • ScanInput opens Scanner widget which uses the device camera for real-time scanning
  • Supports all standard 1D and 2D barcodes
  • Scanned value stored as plain string in inputData

Signature Capture

  • Opens a full-screen drawing canvas (Routes.SIGNATURE)
  • PNG file is saved locally, then uploaded to CDN in the background
  • inputData stores {path, url} — use url for database insertion

Offline Mode / Local SQLite

All condition expressions (visibility_condition, disabled_condition, required_condition), and color_query evaluation are run against a local SQLite database via DatabaseHelper. This means conditions work even without network connectivity.

Pull-to-Refresh

Every view has a refresh button (↻) in the AppBar. Tapping it:

  1. Clears all inputData
  2. Re-fetches all data queries
  3. Re-runs onload actions

Additionally, when the app returns to foreground, screens auto-refresh (controllable per session).

Push Notifications

Sent via notification action with notification_type: "app". FCM (Firebase Cloud Messaging) handles delivery on Android; APNs on iOS.

OTP Verification (🆕 NEW)

The otp action sends a 4-digit OTP via the Fieldproxy Task API and shows a native PIN input dialog with:

  • 2.5-minute countdown timer
  • Resend button (activates after timer expires)
  • Returns true on successful verification, false on failure/dismiss

Deep Linking

External navigation via navigation_type: "url" opens URLs using ChromeSafariBrowser (Custom Chrome Tab on Android, Safari View Controller on iOS).

13. Navigation Patterns

Stack Navigation (Push / Pop)

{
  "type": "navigation",
  "app_id": "myApp",
  "view_id": "job_detail",
  "arguments": {"job_id": "{{jobs.data.[index].id}}"}
}

The destination screen is pushed onto the navigation stack. The AppBar automatically shows a back chevron (←).

Pop and Return Data

{
  "type": "navigation",
  "app_id": "myApp",
  "view_id": "job_list",
  "back": true,
  "arguments": {
    "refresh": ["jobs_list"]
  }
}

When back: true, the current screen is popped. If refresh array is present, the calling screen re-fetches the listed data sources.

Replace Navigation

{
  "type": "navigation",
  "replace": true,
  "app_id": "myApp",
  "view_id": "login_screen"
}

Replaces current screen in the stack (no back navigation).

Arguments Between Views

Arguments passed via navigation are accessible in the destination view as {{arguments.key_name}}. They can be used in:

  • SQL data queries in data{}
  • Template strings anywhere in ui[]
  • Condition expressions

Back Navigation Guard (🆕 NEW)

Prevent accidental back navigation with unsaved form data:

"back_confirmation": {
  "enabled": true,
  "title": "Unsaved Changes",
  "message": "You have unsaved changes. Do you want to discard them?"
}
