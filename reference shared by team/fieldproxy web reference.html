Fieldproxy Web View — Complete Technical Reference

Source: operationalportal.json  |  Format: type: web  |  Version: Full Analysis

1. Root-Level JSON Structure

Every Fieldproxy web app begins with a single root object:

{
  "type": "web",
  "title": "Fieldproxy Clone - Requests Management",
  "app_id": "EuGQxgu",
  "default_view": "home_view",
  "element_id": "root_app",
  "description": "Operational portal for field service management",
  "metadata": { ... },
  "views": {
    "home_view": { ... },
    "jobs_view": { ... }
  }
}

Root Properties

PropertyTypeRequiredPurpose
typestringYesAlways "web" for web apps
titlestringYesApp name shown in the browser/header
app_idstringYesUnique app identifier used in navigation
default_viewstringYesThe view rendered on first load (key under views)
element_idstringOptionalUnique ID for the root element
descriptionstringOptionalHuman-readable description of the app
metadataobjectOptionalApp-level metadata (versioning, org info, etc.)
viewsobjectYesMap of view_id → view definition

Views Dictionary Pattern

"views": {
  "home_view": {
    "ui": [ ... ],
    "data": { ... },
    "view_id": "home_view",
    "title": "Dashboard"
  }
}

Each view is a self-contained screen definition:

PropertyTypeRequiredPurpose
uiarrayYesRoot UI element array
dataobjectOptionalNamed SQL queries loaded on screen init
view_idstringOptionalIdentifies this view
titlestringOptionalScreen title
onloadobjectOptionalActions to execute when the view loads

2. How UI Nesting Works

view
 └── ui: []
      ├── sidebar (type: sidebar)
      └── container (type: container)
           └── ui: []
                ├── row (type: row)
                │    └── ui: []
                │         ├── text
                │         ├── spacer
                │         └── button
                ├── space_widget
                ├── tab_group
                │    └── ui: []
                │         └── table
                └── wrap
                     └── ui: []
                          ├── container
                          └── container

Key rule: The sidebar and the main content are always the two direct children of the view's ui array.

Canonical View Shell

{
  "ui": [
    {
      "type": "sidebar",
      "items": [ ... ],
      "width": 240,
      "background_color": "#1F2937",
      "header": "App Name",
      "header_color": "#FFFFFF",
      "element_id": "sidebar_home"
    },
    {
      "type": "container",
      "width": 12,
      "color": "#FFFFFF",
      "background_color": "#F8FAFC",
      "ui": [
        { "type": "space_widget", "space": 24 }
      ]
    }
  ],
  "data": {
    "my_query": "SELECT ..."
  }
}

Key Pattern: Dynamic Color with color_query

{
  "type": "container",
  "color_query": "CASE WHEN '{{status}}' = 'active' THEN '#D1FAE5' ELSE '#F5F5F5' END AS color"
}
{
  "type": "text",
  "text_color_query": "CASE WHEN '{{status}}' = 'active' THEN '#059669' ELSE '#6B7280' END AS color"
}

Rules:

  • No SELECT keyword — just CASE WHEN … END AS color
  • color_query → applies to container background
  • text_color_query → applies to text foreground
  • Template variables ({{rowData.field}}, {{arguments.field}}, {{query.data.[index].field}}) all work inside the SQL expression
